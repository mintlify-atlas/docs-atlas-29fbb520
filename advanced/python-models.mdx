## Overview

Circulatory Autogen can generate standalone Python models from CellML files using libCellML. Python models offer:

- **Flexibility** - Easy to modify and extend
- **Portability** - No compilation required
- **Moderate performance** - Faster than interpreted CellML, slower than C++
- **Rich ecosystem** - Access to SciPy's solvers and analysis tools

## Enabling Python Generation

In `user_inputs.yaml`:

```yaml user_run_files/user_inputs.yaml
model_type: python
solver: solve_ivp
solver_info:
  method: BDF  # or RK45, Radau, LSODA, etc.
  MaximumStep: 0.0001
```

## The PythonGenerator Class

The generator is implemented in `src/generators/PythonGenerator.py`:

```python src/generators/PythonGenerator.py
class PythonGenerator:
    """
    Generate a Python module from a CellML file.

    Usage:
        gen = PythonGenerator('model.cellml', output_dir='out')
        py_path = gen.generate()
    """

    def __init__(self, cellml_path: str, output_dir: Optional[str] = None, 
                 module_name: Optional[str] = None):
        self.cellml_path = cellml_path
        self.output_dir = output_dir or os.path.dirname(os.path.abspath(cellml_path))
        self.module_name = module_name or os.path.splitext(os.path.basename(cellml_path))[0]
```

### Generation Process

1. **Parse CellML model**
2. **Resolve imports** (handles CellML 1.1 and 2.0)
3. **Flatten model** into single representation
4. **Analyze** to ensure valid ODE structure
5. **Generate Python code** using libCellML's Python profile
6. **Validate** with a test simulation

## Solver Options with solve_ivp

Python models use SciPy's `solve_ivp` with multiple method options.

### Available Methods

```python src/solver_wrappers/__init__.py
solve_ivp_methods = ['RK45', 'RK23', 'DOP853', 'Radau', 'BDF', 'LSODA', 'RK4', 'forward_euler']
```

### Method Selection Guide

| Method | Best For | Order | Adaptive |
|--------|----------|-------|----------|
| **BDF** | Stiff systems (recommended for cardiovascular models) | Variable | Yes |
| **Radau** | Stiff systems, high accuracy | 5 | Yes |
| **LSODA** | Automatic stiff/non-stiff switching | Variable | Yes |
| **RK45** | Non-stiff systems, general purpose | 5 | Yes |
| **DOP853** | High-accuracy non-stiff | 8 | Yes |
| **RK23** | Simple non-stiff | 3 | Yes |

### Configuration in user_inputs.yaml

```yaml user_run_files/user_inputs.yaml
solver: solve_ivp
solver_info:
  method: BDF           # Solver method
  MaximumStep: 0.0001   # Maximum time step (mapped to max_step)
  MaximumNumberOfSteps: 5000  # Not directly used by solve_ivp
  # Optional SciPy-specific parameters:
  rtol: 1e-6           # Relative tolerance
  atol: 1e-8           # Absolute tolerance
  max_step: 0.0001     # Alternative to MaximumStep
```

### SimulationHelper for Python Models

The `PythonSimulationHelper` class wraps solve_ivp:

```python src/solver_wrappers/python_solver_helper.py
class SimulationHelper:
    def __init__(self, model_path, dt, sim_time, solver_info=None, pre_time=0.0):
        self.model_path = model_path
        self.dt = dt
        self.sim_time = sim_time
        self.solver_info = solver_info or {}
        solver_method = self.solver_info.get('method')
        self.solve_ivp_method = solver_method 
        self.solve_ivp_kwargs = {
            k: v
            for k, v in self.solver_info.items()
            if k in ["rtol", "atol", "max_step", "vectorized", "dense_output", "jac"]
        }
```

### Setting Solver Method

```python
sim_helper = PythonSimulationHelper(output_path, dt=0.00001, sim_time=0.00001)
sim_helper.set_solve_ivp_method('BDF')
success = sim_helper.run()
```

### Advanced Solver Options

**Provide Jacobian** for faster stiff solving:

```yaml
solver_info:
  method: BDF
  jac: analytical  # If model provides jacobian function
```

**Dense output** for interpolation:

```yaml
solver_info:
  method: BDF
  dense_output: true
```

**Vector optimization**:

```yaml
solver_info:
  method: BDF
  vectorized: true  # If RHS is vectorized
```

## When to Use Python vs. CellML vs. C++

### Use Python when:

✅ You need moderate performance (2-3× faster than CellML)  
✅ You want to modify model equations programmatically  
✅ You need access to SciPy's advanced solvers (Radau, LSODA)  
✅ You're developing and testing new models  
✅ Portability across platforms is important  

### Use CellML when:

✅ You're starting development and iterating quickly  
✅ You want to visualize and edit models in OpenCOR  
✅ Model validation and debugging is the priority  
✅ Performance is not critical  

### Use C++ when:

✅ You need maximum performance (10-50× faster than CellML)  
✅ You're coupling to 1D finite volume models  
✅ You're running large-scale parameter identification  
✅ You're deploying to HPC systems  
✅ You have long simulation times  

## Validation and Testing

The generator automatically validates generated code:

```python src/generators/PythonGenerator.py
sim_helper = PythonSimulationHelper(output_path, dt=0.00001, sim_time=0.00001)
sim_helper.set_solve_ivp_method('BDF')
success = sim_helper.run()

if success:
    print('Model generation has been successful.')
    return output_path
else:
    raise ValueError('Model generation has failed. Or the simulation fails when trying to simulate'
            'for 0.00001 seconds in Python')
```

This runs a brief test simulation to ensure:
- The model compiles correctly
- Equations are well-formed
- Initial conditions are valid
- The solver can integrate the system

## Troubleshooting

### Import Errors

If you see "libcellml is required to generate Python models":

```bash
pip install libcellml
```

### Model Invalid for Python Generation

Error: "Generated model is not a valid ODE model"

This means the CellML model contains:
- Algebraic constraints not solvable as ODEs
- Circular dependencies
- Undefined variables

Check the model in OpenCOR and fix validation errors.

### Empty Generated Code

Error: "Generated Python code is empty. Model invalid"

The libCellML generator failed. Check:
- CellML model validates in OpenCOR
- All imports are resolved correctly
- No malformed mathematics in equations

### Simulation Fails Immediately

If test simulation fails:

1. **Check initial conditions** - Ensure they're physiologically reasonable
2. **Try different solver** - Some models work better with Radau or LSODA
3. **Adjust tolerances** - Increase `rtol` and `atol` for difficult models
4. **Reduce max_step** - Smaller steps can help with stiff systems

## Performance Optimization

### Solver Selection

For typical cardiovascular models:

```yaml
solver_info:
  method: BDF      # Best for stiff cardiovascular systems
  rtol: 1e-6       # Balance accuracy and speed
  atol: 1e-8
  max_step: 0.001  # Limit step size for stability
```

### Pre-compilation

Python models can be byte-compiled for faster imports:

```bash
python -m py_compile generated_model.py
```
