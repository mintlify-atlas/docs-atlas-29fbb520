After finding optimal parameters with an optimizer, use Markov Chain Monte Carlo (MCMC) to quantify uncertainty and explore the full posterior distribution of parameters.

## What is MCMC?

MCMC samples from the posterior distribution of parameters given the data:

```
P(parameters | data) ∝ P(data | parameters) × P(parameters)
```

- **Likelihood**: `P(data | parameters)` - how well parameters explain data
- **Prior**: `P(parameters)` - parameter bounds from `params_for_id.csv`
- **Posterior**: `P(parameters | data)` - probability distribution of parameters

MCMC explores this distribution using random walks (chains) that preferentially visit high-probability regions.

## When to Use MCMC

Use MCMC after parameter identification to:

- **Quantify uncertainty** in parameter estimates
- **Identify correlations** between parameters
- **Assess identifiability** - are parameters well-constrained?
- **Validate optimization** - check if optimizer found global minimum
- **Report confidence intervals** for publication

<Note>
MCMC is computationally expensive. Run it after optimization has found good initial parameters.
</Note>

### Enable MCMC in user_inputs.yaml

```yaml
do_mcmc: true

mcmc_options:

```

### mcmc_options Parameters

<ParamField path="num_steps" type="integer" required>
  Number of MCMC iterations per walker. More steps = better convergence.
  
  - **Minimum**: ~500 for quick tests
  - **Recommended**: 1000-5000 for production
  - **Burn-in**: First 50% discarded automatically
</ParamField>

<ParamField path="num_walkers" type="integer" required>
  Number of independent MCMC chains (walkers) to run in parallel.
  
  - **Minimum**: 2 × num_parameters (emcee requirement)
  - **Recommended**: 32-128 for good sampling
  - **More walkers**: Better exploration, more expensive
</ParamField>

<ParamField path="cost_type" type="string" default="MSE">
  Cost function type:
  - `MSE`: Mean squared error (equivalent to Gaussian likelihood)
  - `AE`: Absolute error
</ParamField>

<ParamField path="cost_convergence" type="float" default="0.001">
  Convergence tolerance for cost function.
</ParamField>


### Workflow

1. **Optimization** finds best parameters (mode of posterior)
2. **MCMC** explores around best parameters (full posterior)
3. **Analysis** examines chains, posteriors, and correlations

## MCMC Algorithm

### Initialization

Walkers start near the best parameters from optimization:

```python
# Initial positions: small perturbations around best_params
pos = best_params + 1e-4 × np.random.randn(num_walkers, num_params)
```

### Sampling Loop

For each step:

1. **Propose** new parameter values for each walker
2. **Evaluate** log-likelihood for proposed values
3. **Accept/reject** based on Metropolis-Hastings criterion
4. **Store** accepted samples

### Log-Likelihood Function

The likelihood connects cost to probability:

```python
def log_likelihood(params):
    cost = get_cost_from_params(params)
    
    if cost_type == 'MSE':
        # Gaussian likelihood
        return -0.5 × cost
    else:
        return -cost
```

For Gaussian noise, minimizing MSE ≡ maximizing likelihood.

### Log-Prior Function

Uniform prior within parameter bounds:

```python
def log_prior(params):
    for i, param in enumerate(params):
        if param < min[i] or param > max[i]:
            return -np.inf  # Outside bounds
    return 0.0  # Uniform within bounds
```

### Log-Posterior Function

```python
def log_posterior(params):
    lp = log_prior(params)
    if not np.isfinite(lp):
        return -np.inf
    return lp + log_likelihood(params)
```

### Visual Checks

## Best Practices

### Burn-in Period

<AccordionGroup>
  <Accordion title="What is burn-in?">
    Initial steps where chains haven't reached the posterior distribution yet. These samples are biased by starting positions.
  </Accordion>
  
  <Accordion title="How much to discard?">
    - **Default**: 50% (automatic in Circulatory Autogen)
    - **Check trace plots**: Discard samples before convergence
    - **Conservative**: Discard 70-80% for safety
  </Accordion>
</AccordionGroup>

### Number of Samples

**Total samples = (num_steps - burn_in) × num_walkers**

For reliable posteriors:
- **Minimum**: ~1,000 effective samples
- **Recommended**: 10,000+ total samples
- **Example**: 1000 steps × 64 walkers × 0.5 (after burn-in) = 32,000 samples

### Initialization Strategy

**Option 1: From optimizer** (default)
```python
pos = best_params + small_noise
```

**Option 2: Random in bounds**
```python
pos = np.random.uniform(param_mins, param_maxs, 
                        size=(num_walkers, num_params))
```

**Option 3: Previous MCMC run**
```python
pos = previous_chain[-1, :, :]  # Last step of previous run
```

## Common Issues

### Stuck Chains

**Symptom:** Some walkers have constant parameter values

**Cause:** Initialization outside prior bounds or at low-probability regions

**Solution:** Automatic detection removes stuck walkers:
```python
if np.std(walker_samples) == 0:
    remove_walker(walker_idx)
```

### Poor Mixing

**Symptom:** Chains don't overlap, stay in separate regions

**Cause:**
- Too few steps
- Poor initialization
- Multi-modal posterior

**Solution:**
- Increase `num_steps`
- Use more walkers
- Run multiple independent chains

### High Autocorrelation

**Symptom:** Samples are not independent

**Cause:** Slow exploration of parameter space

**Solution:**
- Thin chains (keep every Nth sample)
- Run longer
- Use more walkers

```python
thin = 5
thinned_samples = samples[::thin, :, :]
```

## Advanced Usage

### Custom Likelihood Functions

Define custom likelihoods in `funcs_user/cost_funcs_user.py`:

```python
def custom_log_likelihood(obs_values, model_values, std_values):
    """
    Custom likelihood for non-Gaussian noise.
    
    Args:
        obs_values: Ground truth observations
        model_values: Model predictions
        std_values: Measurement uncertainties
    
    Returns:
        log_likelihood: Log probability
    """
    # Example: Laplace (L1) likelihood
    residuals = np.abs(obs_values - model_values) / std_values
    return -np.sum(residuals)
```

## Output Interpretation

### Well-Identified Parameters

**Corner plot shows:**
- Narrow, symmetric distributions
- Low correlation with other parameters
- 90% CI spans ~1 order of magnitude or less

**Conclusion:** Parameter is well-constrained by data

### Poorly-Identified Parameters

**Corner plot shows:**
- Wide, flat distributions  
- Strong correlations with other parameters
- 90% CI spans multiple orders of magnitude
- Distribution reaches prior bounds

**Conclusion:** Need more/better data to constrain this parameter

### Strong Correlations

**Corner plot shows:**
- Elongated ellipses in 2D plots
- Clear linear relationships

**Interpretation:**
- Parameters trade off (sloppy model)
- Only certain combinations matter
- Consider reparameterization
